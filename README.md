# Phishing Awareness Login Demo

A controlled security awareness demo built for an authorized Practical Hacking course exercise. The project recreates a familiar university-style login experience to demonstrate how convincing phishing pages can look in practice, while keeping the implementation intentionally safe: passwords are never stored, usernames are masked before tracking, and the page reveals the training purpose immediately after interaction.

## Live Demo

[Open the live demo](https://idp-metropolia-profile-cas-login.onrender.com)

## Project Overview

The purpose of this project was to demonstrate a real-world phishing risk in a controlled educational setting. Instead of focusing only on theory, I built an end-to-end demo that shows how strongly users can trust familiar visual design if they do not verify the actual domain, context, and purpose of the page.

This project combines front-end implementation, responsive UI work, backend event tracking, deployment, and practical security awareness design. The result is a realistic but safe demonstration that can be shown clearly in a course or presentation setting.

## Key Features

- Realistic university-style login page recreation
- Responsive front-end for desktop and mobile
- Controlled educational phishing awareness flow
- Passwords are never stored or displayed
- Usernames are masked before tracking
- Lightweight backend tracking for total interactions and recent masked usernames
- Public live deployment for demonstration purposes

## Why this project matters

This project is not about copying a login page for its own sake. It is about demonstrating, in a realistic but controlled way, how visual familiarity can create misplaced trust. From a defensive perspective, this is exactly the kind of practical example that makes phishing risk easier to understand than a purely theoretical explanation.

## Security and Ethics

This demo was created for an authorized Practical Hacking course exercise and intentionally implemented with safe boundaries:

- passwords are not stored
- usernames are masked before tracking
- the user is shown a clear explanation after interaction
- the goal is security awareness, not credential collection
